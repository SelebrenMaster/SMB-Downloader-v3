# Почему проводник быстрее SMB-Downloader?

## Технический анализ разницы в скорости копирования

При копировании большого файла (например, 4 ГБ) с сетевого диска через Проводник Windows (Ctrl+C → Ctrl+V) скорость будет заметно выше, чем через SMB-Downloader. Ниже — причины.

---

## 1. Уровень копирования

| Проводник (Explorer) | SMB-Downloader |
|---|---|
| `CopyFileExW` — системный API ядра Windows | `open()`, `read()`, `write()` — Python, уровень пользователя |
| Асинхронный I/O, DMA | Синхронное чтение/запись, один поток |
| Оптимизировано Microsoft десятилетиями | Ограничено интерпретатором Python |

Проводник копирует **на уровне ядра ОС** — данные идут напрямую от сетевого драйвера к дисковому, без промежуточных копий в память процесса.

Функция `CopyFileExW` использует:
- **Overlapped I/O** — асинхронные операции чтения/записи
- **DMA (Direct Memory Access)** — данные передаются между устройствами без участия CPU
- **Kernel-mode буферы** — данные не копируются в пользовательское пространство

---

## 2. Архитектура выполнения

### Проводник — параллельное выполнение

```
Поток 1 (чтение из сети):  [READ 1]     [READ 2]     [READ 3]     [READ 4]
Поток 2 (запись на диск):            [WRITE 1]    [WRITE 2]    [WRITE 3]
```

Пока один блок записывается на диск, следующий **уже читается** из сети. Операции перекрываются во времени.

### SMB-Downloader — последовательное выполнение

```
[READ 1] → [WRITE 1] → [flush] → [READ 2] → [WRITE 2] → [flush] → ...
```

Каждая операция ждёт завершения предыдущей. Пока идёт запись — сеть простаивает. Пока идёт чтение — диск простаивает.

---

## 3. Накладные расходы SMB-Downloader

Каждый чанк (блок данных) проходит через множество операций:

```python
# 1. Чтение из источника
data = fsrc.read(to_read)              # системный вызов

# 2. Запись в приёмник
fdst.write(data)                       # системный вызов
fdst.flush()                           # !!! принудительный сброс на диск

# 3. Динамический размер блока
self.chunk_sizer.update(speed)         # расчёт средней скорости, выбор блока

# 4. Ограничение скорости
self._throttle(len(data))              # time.sleep() для throttling

# 5. Автосохранение
self._auto_save_check()                # проверка, запись .meta

# 6. Обновление интерфейса
self.progress_cb(...)                  # root.after() → обновление UI
self.status_cb(...)                    # ещё один callback
```

**Самый дорогой элемент — `fdst.flush()`.** Он принудительно сбрасывает данные на диск, не давая ОС буферизировать запись. Без него ОС накапливает данные в памяти и пишет большими блоками — это в разы быстрее.

### Сравнение затрат на один чанк

| Операция | Время (примерно) |
|---|---|
| `fsrc.read()` | 1–5 мс |
| `fdst.write()` | 1–3 мс |
| `fdst.flush()` | **5–50 мс** |
| `chunk_sizer.update()` | 0.1 мс |
| `_throttle()` | 0–100 мс |
| `_auto_save_check()` | 0.1 мс |
| `progress_cb()` | 1–5 мс |
| **Итого на чанк** | **~50–150 мс** |

При размере чанка 256 КБ это даёт пропускную способность ≈ 1.7–5 МБ/с — независимо от скорости сети.

---

## 4. Почему `flush()` так сильно тормозит

Без `flush()`:

```
Python → write() → буфер ОС → (накопление) → запись на диск большими блоками
```

ОС копит данные в памяти (обычно несколько МБ) и пишет одним большим блоком. Это эффективно для HDD и SSD.

С `flush()`:

```
Python → write() → flush() → НЕМЕДЛЕННАЯ запись на диск
```

Каждые 256 КБ записываются отдельно. Для HDD это означает:
- Перемещение головки диска для каждого чанка
- Ожидание поворота пластины
- Накладные расходы файловой системы (обновление MFT/FAT)

---

## 5. Влияние ограничения скорости

Если в настройках установлен лимит скорости (например, 5000 КБ/с), программа намеренно замедляется:

```python
def _throttle(self, bytes_read: int):
    expected = (bytes_read / 1024) / self.speed_limit_kbps
    sleep_time = expected - elapsed
    if sleep_time > 0:
        time.sleep(sleep_time)
```

Но даже при лимите `0` (без ограничений) микрозадержки остаются из-за всех операций, перечисленных в пункте 3.

---

## 6. Сравнение архитектуры

| Параметр | Проводник | SMB-Downloader |
|---|---|---|
| API | `CopyFileExW` (ядро) | `open/read/write` (Python) |
| I/O модель | Асинхронный, overlapped | Синхронный, последовательный |
| Потоки | Несколько (чтение + запись параллельно) | Один |
| Буферизация | Системная, ядро | Ручная, 256 КБ – 8 МБ |
| `flush()` | Нет (ОС решает когда писать) | Да (после каждого чанка) |
| Ограничение скорости | Нет | Есть (`time.sleep`) |
| UI-обновления | Отдельный поток, редкие | `root.after()` каждый 500 мс |
| Автосохранение | Нет | Каждые 30 с / 10 МБ |
| Рассчёт MD5 | Отдельно после копирования | После копирования (в главном потоке) |

---

## 7. Резюме

SMB-Downloader **медленнее** проводника не потому, что «плохо написан», а потому, что он **делает гораздо больше работы**:

1. **`flush()` после каждого чанка** — главная причина потери скорости
2. **Последовательное чтение → запись** — нет параллелизма
3. **Обновление UI** — дополнительные вызовы и контекстные переключения
4. **Автосохранение** — запись метаданных на диск
5. **Динамический размер блока** — вычисления на лету
6. **Ограничение скорости** — намеренные задержки (`sleep`)

Всё это — **осознанный компромисс**. Мы жертвуем скоростью ради:
- Возможности **возобновить** копирование с любой точки
- Точного **прогресса** в реальном времени
- **Надёжности** — данные гарантированно записаны после каждого чанка
- **Контроля** — ограничение скорости, логирование, история
